## Validation Notes

Per [constitution/01-template-alignment.md](../constitution/01-template-alignment.md) seven key rules + the new Requirement Level rule (post-ADR-0012):

1. **One row per inventory type per requirement.** Cross-check: `{req-id}` rows that share Inventory factor + complexity (potential duplicate) → log warning.
2. **Hours never entered manually.** Only VS/S/M/C/VC counts are populated; hours are computed at output time in `Estimation-ModuleBuildHrs.md`.
3. **Solution / Rationale mandatory.** Rows with empty Rationale → emit Typed Gap of category `MISSING-DETAIL`.
4. **Fitment must be one of 8 approved values.** Validated by `/estimate`.
5. **Type of Integration = NA when Fitment != Integration** (Key Rule #7). Validated by `/estimate`.
6. **Requirement Level MUST be one of L1/L2/L3/L4/L5** (per [05-requirement-levels.md](../constitution/05-requirement-levels.md) and [ADR-0012](../../../design/adr/0012-requirement-level-taxonomy.md)). Inferred levels emit Typed Gap `REQUIREMENT-LEVEL-INFERENCE`.

## Export semantics (per [ADR-0012](../../../design/adr/0012-requirement-level-taxonomy.md))

| Surface | Requirement Level | Categorization |
|---|---|---|
| Markdown (this file) | 1 column, value like `L3` | 1 column, path like `acme > Service > Case Mgmt > Case Triage > New CRM Plugin Build & UT` |
| XLSX export | **1 column** `Requirement Level` (NEVER split into 5) | 5 columns: `Project`, `Module`, `Capability`, `Feature`, `Factor` (NEVER labelled L1-L5) |
| CSV export | 1 column `Requirement Level` | 1 column `Categorization` (full path) |
| JSON export | 1 field `requirementLevel: "L3"` | 1 path string OR a nested object `{ project, module, capability, feature, factor }` |
