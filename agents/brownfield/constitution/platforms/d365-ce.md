## Accepted input file types

Under `_brownfield/input/d365-ce/`:

| Input | Format | Notes |
|---|---|---|
| Unmanaged solution ZIP | `.zip` | Preferred; contains `Entities/`, `Workflows/`, `WebResources/`, `Roles/`, `solution.xml` |
| Managed solution ZIP | `.zip` | Allowed; some metadata is stripped (e.g., plugin step bodies) â€” flag `MISSING-INPUT` for stripped portions |
| Plugin assembly | `.dll` | Read assembly metadata only; logic requires source â€” flag `BLOCKED-BY-BINARY` for full logic when source absent |
| Plugin source | `.cs` / `.csproj` | Optional; when present, enables `validate_plugin_logic` |
| JS web resources | `.js` (extracted from `WebResources/`) | Embedded in solution; full source available |
| HTML / XML / image web resources | `.html` / `.xml` / `.png` etc. | Embedded |
| PCF source | `index.ts` / `manifest.xml` / `ControlManifest.Input.xml` | Optional; when present, enables PCF documentation |
| Power Pages site | site XML (zip export) | Optional; treated under platform `power-pages` |

## Scan strategy

1. Unzip solution to `_brownfield/extracted/d365-ce/{solution-name}/`
2. Walk `Entities/*/Entity.xml` â†’ enumerate entities + attributes + relationships + keys
3. Walk `Workflows/*.xml` â†’ enumerate classic workflows + BPFs (distinguished by `<Category>0</Category>` vs `<Category>4</Category>`)
4. Walk `WebResources/` â†’ enumerate per-file (JS, HTML, XML, image)
5. Walk `Roles/*.xml` â†’ enumerate security roles
6. Walk `RibbonDiffXml/` â†’ enumerate ribbon diffs per entity
7. Walk `SiteMap.xml` â†’ enumerate sitemap nodes
8. Walk `Templates/` â†’ enumerate email / Word / Excel templates
9. Walk `PluginAssemblies/` + `SdkMessageProcessingSteps/` â†’ enumerate plugin classes + steps
10. Walk `EntityKeys/` â†’ enumerate alternate keys
11. Walk `Dashboards/`, `Charts/`, `SavedQueries/`, `UserQueries/` â†’ enumerate dashboards, charts, views

## Module detection signals (CE side)

Per [`templates/module-detection.yaml`](../../templates/module-detection.yaml). Module bindings consult inventory for:

- **customer-service**: entities `sla`, `queue`, `routingrule`, `entitlement`, `knowledgearticle`, `msdyn_macro` present
- **sales**: entities `opportunity`, `quote`, `salesorder`, `invoice`, `product`, `pricelevel` present
- **marketing**: entities starting with `msdynmkt_` or `list` present
- **field-service**: entities starting with `msdyn_workorder` present

## Analysis rules

### Entities

- Documented depth: every attribute (count + name + type + required)
- Cross-refs: plugin-steps targeting this entity; forms binding to this entity; views over this entity; flows triggered by this entity
- Alternate keys: list when present

### Plugins

- One doc per plugin **class** (NOT per assembly)
- For each step: message + primary entity + stage + mode (sync/async) + filter attrs + pre/post-image
- Logic narrative requires `validate_plugin_logic` â€” source `.cs` enables full body; `.dll` alone produces a `BLOCKED-BY-BINARY` gap

### JS web resources

- One doc per **function** in a web resource (NOT per file)
- For each function: event binding (form, attribute change, ribbon command) + Xrm.WebApi usage + DOM access (flagged)

### Forms

- Documented depth: tab > section > field structure verbatim
- Per-field: requirement level + visibility rules + label + linked attribute
- Subgrids: target entity + view + filters

### BPFs / classic workflows / business rules

- Documented depth: stages + steps + branching + scope + entity binding
- Business rules: condition + actions on each event

### Security roles

- Per-table privilege matrix: read / write / create / delete / append / append-to / share / assign
- Inherited vs explicit

## Cross-references for CE (mechanically computed)

- entity â†” plugin-step (`step.PrimaryEntityName == entity.LogicalName`)
- entity â†” form (`form.ObjectTypeCode == entity.LogicalName`)
- entity â†” view (`view.ReturnedTypeCode == entity.LogicalName`)
- entity â†” flow (`flow.trigger.entity == entity.LogicalName`)
- entity â†” BPF (`bpf.primaryEntity == entity.LogicalName`)
- web-resource â†” form (form xml references resource id)
- plugin-step â†” plugin-class (plugin-step.AssemblyName + ClassName)
