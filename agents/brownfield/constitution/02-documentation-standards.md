# Documentation Standards — brownfield

## The No-Grouping Rule (machine-enforced)

**Forbidden** in any generated doc:
- `"Other plugins (~12 more)"`
- `"Additional JS files..."`
- `"Remaining views (15 not shown)"`
- Any phrasing that gestures at content without listing it

Machine-enforced by `validate_no_grouping`. Re-attempt prompt: "Author one per-file section for each item; do not summarise."

## One artifact, one document

Every artefact instance gets its own document at the path declared by its binding:

- One entity per `entities/{name}.md`
- One plugin **class** per `plugins/{name}.md` (NOT one doc per assembly with all classes lumped)
- One plugin **step** per `plugin-steps/{name}.md`
- One JS web resource per `js/{namespace}.{function}.md`
- One flow per `flows/{name}.md`
- One Canvas screen as a section within the app's doc; one screen is too small for a doc but each screen is a `## §N` section, not omitted

This is enforced by the binding's `outputPath` resolving to a unique path per instance.

## Heading discipline

- ATX-style headings (`#` ... `######`) only — no Setext
- One `# Title` per doc; subsequent sections start at `##`
- TOC auto-generated by `doc_lint` (when ≥3 `##` sections)
- Section numbering optional but consistent within a doc (use `§N.M`)

## Frontmatter

Every generated doc carries:

```yaml
---
artifact: <artifact-type>
platform: <d365-ce|d365-fo|integration|reporting|power-apps|power-pages|custom-pages>
instance: <unique-name>
source-paths:
  - <file-path>
extracted-at: <ISO 8601>
generated-by: brownfield-engine
binding: <binding-file-path>
pattern: <pattern-name>
validators-run: [<validator-names>]
known-gaps: [<gap-id>...]   # empty if doc passes; populated when retry exhausted
---
```

## Quoting source code / configuration

- SQL: verbatim `<CommandText>` from RDL or `.sql` files — never paraphrase
- X++ / C# / JS / Power Fx: short excerpts in fenced code blocks with language tag; long methods linked to source path with line range
- XML / JSON: pretty-printed; sensitive secrets redacted (`{{REDACTED-SECRET}}`)

## Cross-references

Generated mechanically by `brownfield-engine/cross-ref.ts` from binding `crossRefs` rules. Each link includes:

- Relative path to the target doc
- Type marker (e.g., `[entity:account]`, `[plugin-step:CreateContactStep]`)

## Mermaid diagrams

- Allowed (encouraged for topology, ERD, flow sequence)
- One Mermaid block per logical concern (don't combine system context + sequence in one block)
- `doc_lint` enforces Mermaid-only (no PNG / JPG / `<img>`)

## What goes in the doc body

Every pattern's `## Body` section follows a standard structure (see [`templates/patterns/`](../templates/patterns/)). Common requirements:

- **Identification** — name, namespace, source path, version
- **Structural detail** — fields / steps / actions / columns (depending on pattern)
- **Behavior** — what it does, when it fires
- **Cross-references** — mechanically populated
- **Source evidence** — quoted excerpts where validators require

## File naming

- All docs use kebab-case slugs derived from the artifact instance name
- Spaces in source names → hyphens
- Special characters stripped to ASCII-safe
- Per-platform output trees: `docs-generated/technical/{category}/{name}.md` for technical artefacts; `docs-generated/functional/{category}/{name}.md` for functional artefacts
