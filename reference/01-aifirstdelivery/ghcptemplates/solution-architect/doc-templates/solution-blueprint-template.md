# Operating Principles (MANDATORY)

**DO NOT:**
- Invent new business requirements
- Modify scope
- Include code, formula snippets, or task-level steps
- Reproduce per-feature blueprints verbatim

**ONLY:**
- Synthesise information from the provided specs, plans, and per-feature blueprints
- Produce system-level architecture and design
- Make cross-template dependencies explicit

**MAINTAIN:**
- Traceability: Business Capability → Feature → FR → Component
- Naming consistency with the publisher prefix and Azure resource prefix from sibling constitutions
- A single non-redundant view — deduplicate shared Dataverse tables, integration patterns, and security roles

# Diagram Principles (MANDATORY)

Every section marked with a required diagram **MUST include an embedded Mermaid diagram** in the same generation pass as the prose and tables. Do NOT produce diagrams as a separate or follow-up step.

**Diagram rules:**

- Use Mermaid syntax enclosed in triple-backtick `mermaid` code fences.
- Choose the diagram type specified per section — do not substitute.
- Keep diagrams at architecture level — systems, components, and flows only.
- Apply colour coding per `constitution/03-diagram-standards.md` conventions.
- Label all arrows with the interaction type.
- Use `subgraph` blocks to group related components into logical boundaries.
- Diagrams must be self-contained and readable without the surrounding prose.
- Every diagram must have a descriptive heading immediately above it.
- Mermaid syntax must be syntactically valid — validate mentally before outputting.
