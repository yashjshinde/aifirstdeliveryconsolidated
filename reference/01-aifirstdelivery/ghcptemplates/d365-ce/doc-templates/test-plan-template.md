## 1. Test Strategy

### Approach
Risk-based testing prioritised by business impact and technical complexity.
D365 CE-specific risk areas: plugin execution, Business Rule interactions, security role access, solution import.

> **Scope:** This test plan covers functional testing derived from the specification. Unit tests (Plugin/JS/PCF) are generated by `/implement` alongside the code — they are not part of this document.

### Test Levels
| Level | Purpose | When |
|---|---|---|
| System | Full feature end-to-end in D365 CE | After all tasks implemented |
| Integration | Plugin + Dataverse + external system | After deployment to Test |
| UAT | All FR-NNN scenarios verified by business users | Before go-live |
| Security | Role-based access verified | Before every deployment |
| Regression | No existing features broken | Before every deployment |

## 7. Entry and Exit Criteria

### Entry Criteria (before testing begins at each level)
| Level | Entry Criteria |
|---|---|
| Unit | Code written, solution checker passing |
| Integration | Solution deployed to Test environment |
| System | All integration tests passed |
| UAT | System test sign-off, UAT data seeded, business users briefed |

### Exit Criteria (before moving to next level)
| Level | Exit Criteria |
|---|---|
| Unit | 100% pass, ≥80% coverage |
| Integration | 100% pass, no Critical defects |
| System | 100% pass, no Critical/High defects |
| UAT | Business sign-off, no Critical/High defects open |

## 8. Defect Management

| Severity | Definition | Response Time | Resolution Time |
|---|---|---|---|
| Critical | Feature unusable, data loss risk, security breach | 1 hour | 24 hours |
| High | Core functionality blocked, no workaround | 4 hours | 3 business days |
| Medium | Functionality degraded, workaround available | 1 business day | 1 week |
| Low | Cosmetic, minor inconvenience | 3 business days | Post go-live |

### Defect Workflow

1. QA logs defect in {ALM tool} with severity, module, FR reference, and reproduction steps
2. QA Lead triages and assigns to Developer
3. Developer resolves and marks **Ready for Test**
4. QA Engineer verifies fix — closes or reopens defect
5. Regression testing performed on all related test cases

## 11. Non-Functional Test Coverage

| NFR Category | Test Approach | Tool / Method | Acceptance Criteria |
|---|---|---|---|
| Performance | Measure form load and save response time under normal load | EasyRepro + Azure Load Testing | Per `constitution/11-nfr-targets.md` — e.g., form load < {N}s |
| Scalability | Create {N} records concurrently; verify no throttling | Azure Load Testing | All records created, no API throttle errors |
| Security | Validate FLS profiles, role-based access, audit trail enabled | Manual per security matrix | No hard-coded credentials; FLS enforced; audit trail records changes |
| Reliability | Simulate plugin failure and verify retry / idempotency behaviour | Manual + FakeXrmEasy edge cases | No duplicate records; error surfaced correctly to user |
| Usability | Collect UAT feedback on form navigation and error messages | UAT execution | 90%+ task completion without assistance; < 5 critical usability defects |
