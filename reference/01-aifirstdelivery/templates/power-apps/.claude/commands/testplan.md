Generate a Test Plan and Strategy for a Power Platform feature from an approved spec.

## Usage

```
/testplan {feature-name}
```

## Pre-condition Check

1. Read `specs/{feature-name}/review.md`. If not `APPROVED`, stop.

## Steps

2. Read `constitution/08-testing-standards.md`, `constitution/11-nfr-targets.md`, and `constitution/09-document-generation-rules.md`.
3. Read `specs/{feature-name}/spec.md`.
4. Generate all test cases in memory, grouped into seven suites before writing any file:
   - Canvas App tests
   - Model-Driven App tests
   - Power Automate Flow tests
   - Copilot Studio tests (omit file if no Copilot in scope)
   - Security tests
   - UAT tests
   - Regression tests
   Apply the column structure from `doc-templates/test-case-suite-template.md` for each suite.

5. Write the per-suite test case files using `doc-templates/test-case-suite-template.md`:
   - `docs-generated/{feature-name}/test-cases/canvas.md`
   - `docs-generated/{feature-name}/test-cases/model-driven.md`
   - `docs-generated/{feature-name}/test-cases/flow.md`
   - `docs-generated/{feature-name}/test-cases/copilot.md` *(omit if Copilot not in scope)*
   - `docs-generated/{feature-name}/test-cases/security.md`
   - `docs-generated/{feature-name}/test-cases/uat.md`
   - `docs-generated/{feature-name}/test-cases/regression.md`
   Each file starts with a Suite Summary table, then the full test case table.

6. Write the main plan document using `doc-templates/test-plan-template.md`:
   - `docs-generated/{feature-name}/test-plan-and-strategy.md`
   - §3 contains reference tables only (TC-ID, ALM ID, key columns, priority) — no full detail rows.
   - Each §3 subsection includes the relative file link, e.g., `[test-cases/canvas.md](test-cases/canvas.md)`.
   - Populate the Traceability Matrix (§10) and Test Suite Structure (§7) from the generated test cases.

## What the Test Plan Must Cover

### Test Strategy
- Risk-based: prioritise by user-facing impact and data integrity
- Test levels: Component → Integration → System → UAT → Regression → Accessibility

### Test Scope
- In scope: each FR-NNN, each screen/form, each flow, each Copilot topic
- Out of scope: platform internals (Dataverse engine, Power Automate runtime)
- Constraints: delegation limits, test environment licensing

### Test Types for Power Platform

| Type | What to Test | Tool / Method |
|---|---|---|
| Canvas App — Formula | Power Fx formula logic | Power Apps Test Studio |
| Canvas App — Navigation | Screen-to-screen navigation | Test Studio / Manual |
| Canvas App — Delegation | Large dataset filters work correctly | Manual with test data |
| Model-Driven — Form | Field validation, Business Rules | Manual + EasyRepro |
| Model-Driven — Views | Correct records shown per filter | Manual per persona |
| Flow — Happy Path | Flow runs end-to-end correctly | Test Flow in editor |
| Flow — Error Path | Flow fails gracefully, error handled | Fault injection in editor |
| Copilot — Topic | Correct intent matched and responded to | Test Canvas |
| Copilot — Escalation | User escalated when intent not matched | Test Canvas |
| Security | Persona sees only permitted records/screens | Manual per security role |
| Performance | Screen load, flow duration, delegation boundary vs NFR targets in `11-nfr-targets.md` | Power Apps Monitor + Azure Load Testing |
| Accessibility | Keyboard nav, screen reader, contrast | Manual + axe tool |
| UAT | All FR-NNN verified by business users | UAT environment |
| Regression | Existing features not broken | Test Studio suite |

> **Note — Unit tests** (Power Fx formula tests, Flow unit tests) are developer artefacts generated by `/implement` alongside the code — they are not included in this test plan.

### Test Cases (per FR and per Screen/Flow/Topic)
For every FR-NNN generate:

| TC-ID | FR Ref | Component | Type | Description | Pre-conditions | Steps | Expected Result | Priority |
|---|---|---|---|---|---|---|---|---|
| TC-001 | FR-001 | `scrAccountDetail` | Canvas-Formula | {Scenario} | {Setup} | {Steps} | {Outcome} | High |
| TC-002 | FR-001 | `scrAccountDetail` | Canvas-Navigation | {Scenario} | {Setup} | {Steps} | {Outcome} | Medium |
| TC-003 | FR-002 | `Account Created Flow` | Flow-Happy | {Scenario} | {Setup} | {Steps} | {Outcome} | High |
| TC-004 | FR-002 | `Account Created Flow` | Flow-Error | {Scenario} | {Setup} | {Steps} | {Error handled} | High |

- Minimum 2 test cases per FR (positive + negative)
- Security test cases: one per persona per sensitive screen
- Add an `ALM ID` column to every test case table, initialised to `*(pending — set by /alm sync)*`

### Test Case IDs
- Format: `TC-{TYPE}-{NNN}` where TYPE is: `CA` (Canvas), `MD` (Model-Driven), `FL` (Flow), `CO` (Copilot), `SEC` (Security), `U` (UAT), `R` (Regression)
- Numbers are sequential within each type, starting at 001

### Test Data Requirements
- Sample records per entity required for each test scenario
- PII masking approach for test environments
- Delegation test data: must have > 500 records for delegation boundary tests

### Test Environment Strategy
| Test Type | Environment | Notes |
|---|---|---|
| Component / Unit | Developer environment | Personal env |
| Integration | Test environment | Flows and connectors active |
| UAT | UAT environment | Business users, managed solution |
| Regression | Test environment | Automated Test Studio suite |

### Exit Criteria
- No Critical/High defects open
- UAT: all scenarios signed off by named business users
- Accessibility: zero WCAG 2.1 AA failures
- Regression: 100% pass rate
- Security: all personas tested, no unauthorised access found

### Test Suite Structure
After generating all test cases, count by component type and populate the Test Suite Structure table (§7) with totals and priority distribution.

### Test Deliverables
Generate the Test Deliverables table (§8) listing all outputs, owners, and milestones.

### NFR Test Coverage
Populate the NFR Test Coverage section (§9) using `constitution/11-nfr-targets.md` — cover Performance, Scalability, Security, Reliability, Usability, Accessibility.

### Traceability Matrix
Build FR → TC mapping and populate §10. Every FR-NNN must appear at least once.

### Test Execution Summary
Initialise §11 with all component types and zero pass counts.
