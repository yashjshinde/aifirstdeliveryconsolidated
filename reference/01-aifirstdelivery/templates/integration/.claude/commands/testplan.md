Generate a Test Plan and Strategy for an Azure Integration feature from an approved spec.

## Usage

```
/testplan {feature-name}
```

## Pre-condition Check

1. Read `specs/{feature-name}/review.md`. If not `APPROVED`, stop.

## Steps

2. Read `constitution/06-error-handling.md`, `constitution/11-nfr-targets.md`, and `constitution/09-document-generation-rules.md`.
3. Read `specs/{feature-name}/spec.md`.
4. Generate all test cases in memory, grouped into six suites before writing any file:
   - Contract / Schema tests
   - Integration / End-to-End tests
   - Performance tests
   - Security tests
   - Chaos / Resilience tests
   - UAT tests
   Apply the column structure from `doc-templates/test-case-suite-template.md` for each suite.

5. Write the per-suite test case files using `doc-templates/test-case-suite-template.md`:
   - `docs-generated/{feature-name}/test-cases/contract.md`
   - `docs-generated/{feature-name}/test-cases/integration.md`
   - `docs-generated/{feature-name}/test-cases/performance.md`
   - `docs-generated/{feature-name}/test-cases/security.md`
   - `docs-generated/{feature-name}/test-cases/chaos.md`
   - `docs-generated/{feature-name}/test-cases/uat.md`
   Each file starts with a Suite Summary table, then the full test case table.

6. Write the main plan document using `doc-templates/test-plan-template.md`:
   - `docs-generated/{feature-name}/test-plan-and-strategy.md`
   - §3 contains reference tables only (TC-ID, ALM ID, key columns, priority) — no full detail rows.
   - Each §3 subsection includes the relative file link, e.g., `[test-cases/contract.md](test-cases/contract.md)`.
   - Populate the Traceability Matrix (§10) and Test Suite Structure (§7) from the generated test cases.

## What the Test Plan Must Cover

### Test Strategy
- Risk-based approach: prioritise by data integrity risk and business criticality
- Test levels: Unit → Component → Contract → Integration → End-to-End → Performance → Chaos

### Test Scope
- In scope: each FR-NNN and each interface from the Interface Catalogue
- Out of scope: source/target system internal behaviour
- Dependencies: test stubs/mocks required for unavailable systems

### Test Types for Azure Integration

| Type | What to Test | Tool |
|---|---|---|
| Contract Tests | Message schema conformance per interface catalogue | JSON Schema validator |
| Component Tests | Function + Service Bus end-to-end in isolation | Test environment |
| Integration Tests | Full flow A→B with real systems | Integration test env |
| UAT | End-to-end integration scenarios verified by business users | UAT environment |
| Error & Retry Tests | DLQ, retry exhaustion, circuit breaker | Fault injection |
| Performance Tests | Throughput, latency under load | Azure Load Testing |
| Security Tests | Auth, Managed Identity, Key Vault access | Manual + tooling |
| Chaos Tests | Service unavailability, network partition | Azure Chaos Studio |

### Test Cases (per FR and per Interface)
For every FR-NNN and Interface ID, generate:

| TC-ID | FR/IF Ref | Type | Description | Pre-conditions | Steps | Expected Result | Priority |
|---|---|---|---|---|---|---|---|
| TC-001 | FR-001 | Integration | {happy path end-to-end} | {setup} | {steps} | {message received, processed} | Critical |
| TC-002 | FR-001 | Error | {source sends invalid message} | {setup} | {steps} | {DLQ, alert fired} | Critical |
| TC-003 | FR-001 | Performance | {peak load — N msgs/min} | {load gen setup} | {steps} | {< SLA latency} | High |

- Minimum 3 test cases per FR: happy path, error/DLQ path, retry/idempotency path
- Include DLQ replay test for every async flow

> **Note — Unit tests for Azure Functions** (xUnit, Moq) are developer artefacts generated by `/implement` alongside the code. They are not included in this test plan.

### Test Case IDs
- Format: `TC-{TYPE}-{NNN}` where TYPE is: `C` (Contract), `I` (Integration), `P` (Performance), `SEC` (Security), `CH` (Chaos), `U` (UAT)
- Add an `ALM ID` column to every test case table, initialised to `*(pending — set by /alm sync)*`

### Test Data and Stubs
- Test message payloads per schema version
- Stub/mock services needed where target system not available in test environment
- PII masking approach for test data

### Test Environment Strategy
| Test Type | Environment | Data Requirements |
|---|---|---|
| Unit | Local / CI | None |
| Component | Dev environment | Service Bus namespace |
| Integration | Test environment | All systems connected |
| Performance | Dedicated perf env | Load generator configured |

### Exit Criteria
- No Critical/High defects open
- All contract tests passing
- Performance tests meet SLA targets in `constitution/11-nfr-targets.md`
- DLQ replay verified on at least one flow
- UAT: business sign-off on end-to-end integration scenarios

### Test Suite Structure
After generating all test cases, count by test type and populate the Test Suite Structure table (§7) with totals and priority distribution.

### Test Deliverables
Generate the Test Deliverables table (§8) listing all outputs, owners, and milestones.

### NFR Test Coverage
Populate the NFR Test Coverage section (§9) using `constitution/11-nfr-targets.md` — cover Performance, Scalability, Security, Reliability, Availability.

### Traceability Matrix
Build FR/Interface → TC mapping and populate §10. Every FR-NNN and Interface ID must appear at least once.

### Test Execution Summary
Initialise §11 with all test types and zero pass counts.
