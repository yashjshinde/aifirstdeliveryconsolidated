# Project Layout & Scaffolding

> Per-project state lives under `projects/{name}/`. The scaffolder is a PowerShell toolkit at `tools/scaffold/`.

## Scaffolder commands

```powershell
# Create new project skeleton
.\tools\scaffold\New-Project.ps1 -Name "acme-d365" -Agents d365-ce,integration,alm

# Add a feature to an agent in a project
.\tools\scaffold\New-Feature.ps1 -Project "acme-d365" -Agent d365-ce -Feature "case-management"

# Add a brand-new agent to the repo (rare; seeds from constitution/_reference/ + templates/_reference/)
.\tools\scaffold\New-Agent.ps1 -Name "my-new-agent"
```

## `project.config.yaml` (one file controls everything project-wide)

Validated against `schemas/project-config.v1.json`. See [01-repo-structure.md](01-repo-structure.md) for the full schema example.

## Per-feature layout — branches on `docScope`

Per [ADR-0006](adr/0006-doc-scope-domain-vs-feature.md), `New-Feature.ps1` reads the agent's `docScope` from `agents.yaml` and lays out one of two patterns:

### Pattern A — Domain-scoped agents (CE / Integration / Reporting)

Domain-level FDD/TDD/blueprint live at the agent root. Per-feature folder holds only spec / plan / test-plan / reviews / tasks / output.

```
projects/{p}/{agent}/
├── fdd.md                 # DOMAIN — created on first /fdd; grows over time
├── tdd.md                 # DOMAIN
├── blueprint.md           # DOMAIN
├── fdd-assets/            # (CE: HTML mockups per entity)
├── work-items.yaml        # domain-level traceability
├── _handoffs/             # outgoing handoffs to other agents
└── features/
    └── {feature-slug}/
        ├── .workflow.json # phase=DEFINE, gates=all PENDING initially
        ├── spec.md
        ├── plan.md
        ├── test-plan/     # multi-doc folder
        ├── reviews/       # spec-review.md + clarify-report.md + validate-report.md (per ADR-0001)
        ├── tasks/
        └── output/        # code artifacts
```

### Pattern B — Feature-scoped agents (F&O)

Each feature has its own full document set.

```
projects/{p}/d365-fo/
├── work-items.yaml
├── _handoffs/
└── features/
    └── {feature-slug}/
        ├── .workflow.json
        ├── spec.md
        ├── plan.md
        ├── fdd.md         # FEATURE-level
        ├── tdd.md         # FEATURE-level
        ├── blueprint.md   # FEATURE-level
        ├── test-plan/
        ├── reviews/
        ├── tasks/
        └── output/
```

`New-Feature.ps1` reads `agents.yaml docScope` and lays out the correct skeleton. `/fdd` / `/tdd` / `/blueprint` branch on the same key at runtime (see [04-workflow-gates.md](04-workflow-gates.md)).

## Aggregator output (project-level, not per-feature)

```
projects/{p}/_aggregator/
├── architecture/
│   ├── solution-blueprint.md
│   ├── solution-review-report.md
│   └── solution-prototype/             # clickable HTML
└── estimation/
    ├── inputs/                         # drop RFP / requirements docs here
    ├── Estimation-BusinessReqDetail.md
    ├── Estimation-ModuleBuildHrs.md
    ├── Estimation-ModuleOverallHrs.md
    ├── Estimation-Proposed-Factors.md  # conditional
    └── Estimation.xlsx                 # optional, when --export xlsx
```

## Brownfield project layout (only when `mode: brownfield`)

```
projects/{p}/_brownfield/
├── input/                              # raw extracted code / zips / docs
├── inventory.json                      # /scan output (schema: brownfield-inventory.v1)
├── impact-map.json                     # /handoff output for domain agents
├── gap-log.{json,md}                   # the one review artefact (per ADR-0007)
├── coverage-report.md                  # auto-generated by coverage-tracker
└── docs-generated/                     # rendered per-artifact + synthesis docs
```

## Per-feature template-override / constitution-override (optional, per project)

Per [01-repo-structure.md](01-repo-structure.md) two-layer file resolution:

```
projects/{p}/{agent}/features/{f}/
├── templates-override/                 # OPTIONAL — wins over agent default per file
│   ├── fdd.template.md
│   ├── tdd.template.md
│   └── ...
└── constitution-override/              # OPTIONAL — wins over agent default per file
    ├── 02-nfr.md
    ├── 03-security.md
    └── ...
```

If a file exists at the project level, it fully replaces the agent's version. No merging.

## Hierarchy declared in `project.config.yaml`, not hardcoded

Per [agents/alm.md](agents/alm.md), `alm.hierarchy` declares the project's ALM hierarchy (ADO: Epic / Feature / User Story / Task; JIRA: Initiative / Epic / Story / Sub-task). Plan output's L1-L4 follows this hierarchy via `priorityMap` and `hierarchy` config.
