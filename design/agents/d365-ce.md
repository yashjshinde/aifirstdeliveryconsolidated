# d365-ce — fat agent

> The "fat agent." Owns **everything** on the Dynamics 365 Customer Engagement and Power Platform side: model-driven CE, Canvas apps, Power Pages portal, PCF controls, all Power Automate (cloud + desktop), plus plugins / JS web resources / business rules / BPF / classic workflows / BPM.

## Scope

The agent boundary is set by user goal 1c: *everything* on the Dynamics 365 CE / Power Platform side. Specifically:

- Model-driven entities (tables), columns, relationships
- Forms, views, charts, dashboards, custom pages
- Business rules, business process flows (BPF), classic workflows
- Plugins (sync/async), custom workflow activities, JavaScript web resources, HTML/XML/image web resources
- PCF controls (TypeScript)
- Canvas apps (Power Fx, connectors, components, component libraries)
- Power Pages site, web pages, web templates, web roles, table permissions, web files, content snippets, identity providers
- Power Automate cloud flows + desktop flows (when CE-bound or standalone)
- Solutions, environment variables, connection references
- Security roles, field-level security (FLS), hierarchy security
- Azure Function fallback for complex plugins (handed off to integration agent — but the CE-side request originates here)

## docScope

Per [ADR-0006](../adr/0006-doc-scope-domain-vs-feature.md): **domain** for `fdd`, `tdd`, `blueprint`. One `projects/{p}/d365-ce/fdd.md` per project, growing across features with `feature-id`-tagged sections + rows. Spec, plan, test-plan, reviews, tasks remain feature-scoped under `projects/{p}/d365-ce/features/{f}/`.

## Project-level config absorbed from `project.config.yaml`

- `publisherPrefix` (e.g., `acme`)
- `solutionName`
- `multilingual` per channel — `{ crm: false, portal: true, canvas: false }`
- `oobOverrides` — e.g., `{ businessRules: "prefer-js" }`, `{ complexPlugin: "prefer-azure-function" }`
- `unitTestPolicy` — which artefact types require tests (default: plugin=required, js=required, pcf=required, canvas=optional, portal=required)

## `/fdd` assembly flow (domain mode)

1. Read `agents.yaml` → `docScope.fdd = domain`. Resolve target: `projects/{p}/d365-ce/fdd.md`.
2. If target does not exist → bootstrap from `fdd/_index.template.md` with empty feature content.
3. Read `project.config.yaml` → in-scope sub-platforms (`modelDriven`, `canvas`, `powerPages`, `pcf`, `powerAutomate`). For each sub-platform in scope, load its pack and merge into the `_index` slots.
4. For the current feature, add or update sections in the existing domain FDD:
   - **Structural sections** (new process, scenario, entity, sub-platform area) → append self-contained sub-section with `<!-- feature-id: {feature-slug} -->` marker at the top.
   - **Table-shaped sections** (validation rules, views, security roles, FLS profiles, OOB-first decisions, NFR mapping) → append rows with hidden `feature-id` column.
5. Update cross-doc sections: §1.5 Open Questions roll-up, §10 Handoffs, Appendix B User Story Reference.
6. For each entity in §4.1 Forms Wireframes (model-driven only), invoke `fdd-helpers/form-mockup-generator.prompt.md` with the entity's form spec. Output: interactive HTML at `projects/{p}/d365-ce/fdd-assets/mockups/{entity-form}.html`. Link from §4.5 of the FDD markdown.
7. Re-generate TOC + global traceability matrix.
8. Run inline self-check against `fdd-review.checklist.md` (per [ADR-0001](../adr/0001-review-scope-spec-only.md)). Append "Quality self-check" appendix with findings. BLOCKER findings fail the write.

### Concurrency

Disjoint section-blocks across features don't conflict; appended rows don't conflict. If two features touch the same section, `doc_lint` flags it and the user resolves. No automatic three-way merge.
